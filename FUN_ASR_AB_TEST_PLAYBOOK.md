# Fun-ASR Realtime A/B Test Playbook

这份文档用于验证当前主实时转写链路和 `fun-asr-realtime-2026-02-28` 在你的真实面试场景下谁更好。

适用前提：
- 面试官和候选人分别只有一个说话人。
- 当前主链路建议先用 `Alibaba Paraformer`。
- Fun-ASR 作为 shadow compare provider 运行，不替代主链路。

## 1. 当前实现状态

你现在已经具备下面这些能力：
- 会中主界面有 `Fun-ASR` 按钮，点击后打开右侧 compare 面板。
- compare 面板会展示当前主模型 vs Fun-ASR 的实时转写结果。
- 会中可以直接编辑热词表并保存。
- 热词表会持久化保存，应用重启后仍保留。
- Paraformer 和 Fun-ASR 共用同一份 glossary 文本，但会分别同步成各自模型专属的 `vocabulary_id`。

注意：
- 这不是两个模型共用同一个 `vocabulary_id`。
- 热词更新默认从“下一句”开始生效，不保证修改后立刻影响当前正在说的那一句。

## 2. 测试前准备

每次正式对比前，先确认下面几项：

1. 在设置页确认 `Alibaba STT API Key` 已配置。
2. 当前主 STT provider 设为 `Alibaba Paraformer`。
3. 会中主界面右下区域能看到 `Fun-ASR` 按钮。
4. 点击 `Fun-ASR` 后，右侧能打开 compare 面板。
5. compare 面板里没有 `Alibaba STT API key is required` 之类的错误。
6. 热词表里先放一批你本轮要测的技术词。

## 3. 推荐测试顺序

建议分 6 轮测，不要一上来混在一起。

### A. 基线轮

目标：先看两个模型在普通中文技术面试里的总体稳定性。

操作：
1. 打开 `Fun-ASR` compare 面板。
2. 开始一段 3 到 5 分钟的正常模拟面试。
3. 不改热词，不刻意说术语，不读长串数字。
4. 结束后先看 compare 面板，再去设置页导出 STT benchmark。

重点观察：
- 谁的整句可读性更好。
- 谁的断句更自然。
- 谁的 final 结果更稳定，不会频繁来回抖动。
- 谁对中文口语停顿和重复词处理得更好。

### B. 中英混说术语轮

目标：看中文句子里夹英文技术词时谁更稳。

建议话术：
- “我们现在做的是一个 agent workflow，里面会有 tool calling 和 RAG。”
- “这个服务是 Java Spring Boot，底层用 MySQL Redis 和 Kafka。”
- “我会把 tracing 和 metrics 接到 API Gateway 后面。”

重点观察：
- 英文术语是否被音译成中文。
- 大小写缩写是否能保住，比如 `MCP`、`RPC`、`JVM`。
- 同一术语是否前后写法一致。

### C. 数字与版本轮

目标：看数字、版本号、时间和端口类信息谁更可靠。

建议话术：
- “我们线上是 JDK 21，Spring Boot 3.3，Redis 7，Kafka 3.8。”
- “服务响应时间从 320 毫秒降到 90 毫秒。”
- “这个接口跑在 8080 端口，峰值 QPS 大概 1200。”
- “MySQL 单表数据量大概两千万。”

重点观察：
- 数字是否丢失。
- 版本号是否错位。
- 中文数字和阿拉伯数字的可读性谁更好。
- `3.3`、`8080`、`1200` 这种值是否被错误切分。

### D. 热词编辑轮

目标：验证“会中编辑热词表”是否真的对后续 utterance 生效。

操作：
1. 开会前不要加入 `ThreadLocal`、`CompletableFuture`、`Event Sourcing`。
2. 故意先说一轮，记录两个模型的错误情况。
3. 打开 compare 面板，加入这些热词并点击保存。
4. 等保存成功提示出现。
5. 重新说同一批词，但放在新的句子里。

预期结果：
- 保存后不要求修正当前正在说的那一句。
- 从下一句开始，Paraformer 和 Fun-ASR 都应该更容易命中这些词。
- compare 面板和设置页里的 glossary 应保持同一份内容。

### E. 快速追问轮

目标：看语速快、句子短、连续追问时谁更适合真实面试。

建议话术：
- “为什么这么设计？”
- “如果 Redis 挂了怎么办？”
- “你怎么保证一致性？”
- “这个方案的 trade-off 是什么？”

重点观察：
- 首包速度谁更快。
- 短句 final 结果谁更快落稳。
- 谁更不容易漏字。

### F. 噪声与打断轮

目标：模拟真实环境里咳嗽、停顿、重说、耳机回声。

操作：
1. 人为加入短暂停顿、重说和自我修正。
2. 尝试让面试官音频略带外放感或环境声。
3. 观察两个模型在 partial 到 final 的修正质量。

重点观察：
- 错词是否会在 final 阶段被纠正回来。
- 噪声出现时是否误触发很多无意义文字。
- 回声环境下哪个模型更稳。

## 4. 每轮都要记录的指标

建议不要只看“感觉哪个好”，而是按下面几项记：

### 精准度
- 技术术语是否正确。
- 中英混合词是否保真。
- 数字和版本号是否准确。
- 是否漏词、吞词、重复词。

### 可读性
- 断句是否自然。
- 标点是否合理。
- 最终稿是否接近可以直接阅读的会议文本。

### 延迟
- partial 出现是否快。
- final 稳定是否快。
- 快速追问时是否能跟上。

### 热词效果
- 热词保存后是否对后续句子生效。
- 两个模型是否都受同一份 glossary 文本影响。
- 哪个模型从热词中获益更明显。

### 稳定性
- compare 面板打开后是否持续刷新。
- 会中多次保存热词后是否仍稳定。
- 关闭 compare 面板后是否能正常退出 shadow compare。

## 7. 切主模型的判定标准

只有同时满足下面大部分条件，我才建议把 Fun-ASR 升成主模型：

1. 在术语准确率上明显优于当前 Paraformer。
2. 在数字、版本号、缩写上没有明显退步。
3. 首包和 final 延迟没有显著恶化。
4. 热词编辑后，收益稳定且可重复复现。
5. 连续 20 分钟以上的真实面试里，没有明显稳定性问题。

如果出现下面情况，先不要切主：
- 普通中文句子更准，但数字和版本号明显更差。
- 热词收益不稳定。
- partial 很快，但 final 经常回改成错误结果。
- 面板对比看起来好，但真实 transcript 可读性没有提升。

## 8. 推荐的最终决策方式

建议你至少做 3 场真实对比：

1. Java 后端面试
2. 前端或全栈面试
3. 高频追问、语速偏快的行为面试

然后按下面方式给结论：

- 如果 Fun-ASR 在 3 场里有 2 场以上明显更好，且没有数字/版本号硬伤，可以考虑切主。
- 如果两者接近，但 Fun-ASR 只是在术语轮更好，那就保留 Paraformer 做主，把 Fun-ASR 继续做 compare 观察。
- 如果 Paraformer 在数字、术语、稳定性上更均衡，就继续以 Paraformer 为主，保留 Fun-ASR 作为实验入口。

## 9. 本轮实现对应的预期行为

你在联调时看到下面现象，都算正常：

- 打开 compare 面板后，meeting 已连接时会自动启动 compare session。
- 关闭 compare 面板后，如果这次 compare 是面板自动启动的，会自动停止。
- 保存 glossary 后，主链路和 Fun-ASR 都会刷新到新配置。
- Paraformer 和 Fun-ASR 会各自使用自己的 vocabulary id。
- 设置页和会中面板编辑的是同一份 glossary 源数据。

## 10. 建议你先跑的最小验证闭环

如果你只想先快速验证一次，按这个顺序：

1. 主模型设成 `Alibaba Paraformer`。
2. 打开 compare 面板。
3. 说 5 句中英混合技术话术。
4. 加 3 个新的热词。
5. 再重复说一次同类句子。
6. 观察热词保存前后，Fun-ASR 和 Paraformer 谁提升更明显。
7. 导出 benchmark 报告，留一份测试记录。

做完这 7 步，你就能知道这条链路是不是值得继续投入。
